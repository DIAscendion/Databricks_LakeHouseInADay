# Databricks Silver Layer Physical Data Model

## 1. Overview

This document defines the physical data model for the Silver layer in the Medallion architecture for the TMS (Transportation Management System) Shipment application. The Silver layer stores cleansed, conformed, and enriched data, ready for analytics and downstream processing. It includes error and audit tables for robust data governance.

## 3. Data Retention Policies

### 3.1 Retention Periods for Silver Layer
- **Active Data**: 2 years from load_date
- **Error Data**: 3 years from error_timestamp
- **Audit Data**: 3 years from load_date

### 3.2 Archiving Strategies
- Data older than retention period is moved to a cold storage location (e.g., `/mnt/archive/silver/`)
- Use Databricks jobs to automate archival and deletion
- Maintain Delta Lake time travel for 30 days for rollback and recovery

|---------------------|-------------------|----------------------|-------------------|----------------------------------------------|
| sv_shipment         | SHIPMENT_ID       | sv_audit             | One-to-Many       | Each shipment can have multiple audit logs   |
| sv_shipment         | SHIPMENT_ID       | sv_shipment_error    | One-to-Many       | Each shipment can have multiple error logs   |
| sv_audit            | execution_id      | sv_shipment          | Many-to-One       | Audit logs reference shipment executions     |
| sv_shipment_error   | record_id         | sv_shipment          | Many-to-One       | Error logs reference shipment records        |

## 5. Design Decisions and Assumptions

- All columns from Bronze layer are retained in Silver layer for lineage and traceability.
- Added `id` fields (BIGINT) as surrogate keys for Silver tables.
- Partitioned main table by `CREATED_DTTM` for query performance.
- Error and audit tables are partitioned by timestamp fields.
- No PK/FK constraints are enforced (Databricks/SparkSQL limitation).
- All tables use Delta Lake format for ACID compliance and time travel.
- Metadata columns (`load_date`, `update_date`, `source_system`) are included in all tables.
- Data retention and archiving are managed via Databricks jobs and Delta features.
